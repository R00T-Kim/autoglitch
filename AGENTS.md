# Repository Guidelines

## Project Structure & Module Organization
- `src/` holds core runtime code by domain: `orchestrator/`, `optimizer/`, `hardware/`, `observer/`, `classifier/`, `mapper/`, `llm_advisor/`, shared `types.py`, and utilities.
- `configs/` stores YAML experiment and target configuration (Hydra-style defaults).
- `tests/` is split into `tests/unit/` and `tests/integration/`.
- `data/` contains `raw/`, `processed/`, and `datasets/`; `experiments/` contains configs/logs/results.
- `docs/ARCHITECTURE.md` and `paper/` hold design and publication artifacts.

## Build, Test, and Development Commands
- `python -m venv .venv && source .venv/bin/activate` — create/activate local environment.
- `python -m pip install -e ".[dev]"` — install package plus lint/type/test tools.
- `pytest` — run all tests quickly (default addopts: `-ra`).
- `pytest tests/unit -q` — run unit tests only.
- `pytest --cov=src --cov-report=term-missing --cov-report=xml` — optional coverage run (requires `pytest-cov`).
- `ruff check src tests` / `ruff format src tests` — lint and format Python code.
- `mypy src` — static type check project modules.

## Coding Style & Naming Conventions
- Target Python 3.10+, 4-space indentation, and max line length 100 (Ruff config).
- Use explicit type hints on public methods and dataclasses for structured payloads.
- Naming: modules/files `snake_case`, classes `PascalCase`, constants/enums `UPPER_SNAKE_CASE`.
- Keep module boundaries clean: interface abstractions in `*/base.py`, shared contracts in `src/types.py`.

## Testing Guidelines
- Framework: `pytest` with `pytest-cov`; coverage reports are generated by default.
- Name tests `test_<unit>.py`; mirror source structure where practical (for example, `src/optimizer/bayesian.py` → `tests/unit/test_bayesian.py`).
- Mock hardware/serial dependencies in unit tests; reserve real device workflows for `tests/integration/`.
- Add regression tests for bug fixes and control-loop behavior changes.

## Commit & Pull Request Guidelines
- Git history is not present in this workspace snapshot; use Conventional Commit style (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`).
- Keep commits atomic and scoped to one subsystem.
- PRs should include: concise summary, changed paths, validation output (`pytest`, `ruff`, `mypy`), config deltas (`configs/*.yaml`), and plots/log excerpts when experiment results change.

## Security & Configuration Tips
- Never commit secrets, `.env*`, firmware binaries, or hardware-specific credentials.
- Keep large/generated outputs (`data/raw/`, `experiments/results/`, `mlruns/`) out of version control.
- Use local, untracked config overrides for serial ports and lab-specific target settings.
